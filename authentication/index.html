Authentication

Secure your tools with built-in username/password authentication and session management.

Basic Usage

Add the auth parameter with username/password pairs: 

from func_to_web import run

def admin_dashboard(action: str, value: int):
    return f"Action '{action}' executed with value: {value}"

run(
    admin_dashboard,
    auth={
        "admin": "secure_password_123",
        "user": "another_password"
    }
)

Users must log in before accessing the interface.

Session Management

Sessions are automatically managed with secure, signed cookies: 

run(
    my_function,
    auth={"admin": "password"},
    secret_key="your-secret-key-here"  # Optional
)

  • Auto-login: Users stay logged in (default: 2 weeks)
  • Logout: Button appears in header when authenticated
  • Stateless: No database required

Multiple Functions

Protect multiple tools with a single login: 

from func_to_web import run

def payroll(): 
    return "Payroll Data"

def reports(): 
    return "Q4 Reports"

run(
    [payroll, reports],
    auth={"admin": "admin_pass", "analyst": "finance_pass"}
)

Security Features

Built-in protections:

  • ✅ Timing attack protection: secrets.compare_digest
  • ✅ Signed sessions: Cryptographically signed cookies
  • ✅ CSRF protection: SameSite='Lax' cookies

Limitations:

  • ⚠️ No rate limiting: Use reverse proxy if exposing to public internet
  • ⚠️ No audit logs: Doesn't track user actions
  • ⚠️ No 2FA/password policies: Simple authentication only

Password Security

With a strong, unique password (16+ characters, random), brute-force attacks are mathematically impossible - even at 1 million attempts/second, it would take trillions of years. 

# ❌ Weak - crackable in minutes
auth={"admin": "admin123"}

# ✅ Strong - mathematically unbreakable
auth={"admin": "Tr$9mK#pL2@vN8^qX5&wR7!"}

Generate strong passwords: 

python -c "import secrets; print(secrets.token_urlsafe(24))"
# Example output: "xK9_mPvL8-qR5wN2tY7sX4zM"

Important: Use unique passwords (not reused from other sites). If a password leaks elsewhere, attackers will try it here first.

HTTPS Requirement

Always use HTTPS when accessing over a network (even internal). Without HTTPS, session cookies can be intercepted: 

Scenario without HTTPS:
1. User logs in on public WiFi
2. Attacker on same network captures traffic
3. Session cookie stolen → attacker has full access (no password needed)

Solution: Enable HTTPS (via Nginx or direct SSL): 

# Option 1: Via Nginx (recommended)
run(my_function, host="127.0.0.1", auth={...})
# Let Nginx handle SSL

# Option 2: Direct SSL
run(
    my_function,
    auth={...},
    ssl_keyfile="./key.pem",
    ssl_certfile="./cert.pem"
)

Deployment Options

Internal Network

run(
    my_function,
    host="0.0.0.0",
    auth={"admin": "strong_unique_password"},
    secret_key=os.environ["SECRET_KEY"]
)
Secure if: Strong password + HTTPS + trusted network

Behind VPN (Recommended)

run(
    my_function,
    host="100.64.x.x",  # Tailscale/WireGuard IP
    auth={"admin": os.environ["ADMIN_PASSWORD"]},
    secret_key=os.environ["SECRET_KEY"]
)
Secure if: Strong password + VPN encryption

Public Internet

# App on localhost only
run(
    my_function,
    host="127.0.0.1",
    auth={"admin": os.environ["ADMIN_PASSWORD"]},
    secret_key=os.environ["SECRET_KEY"]
)

Add Nginx with rate limiting: 

server {
    listen 443 ssl http2;
    server_name myapp.example.com;

    ssl_certificate /etc/letsencrypt/live/myapp.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.example.com/privkey.pem;

    location / {
        proxy_pass http://localhost:8000;
        limit_req zone=login burst=5 nodelay;
    }
}

http {
    limit_req_zone $binary_remote_addr zone=login:10m rate=10r/m;
}

Secure if: Strong password + HTTPS + rate limiting

Best Practices

1. Strong, unique passwords 

python -c "import secrets; print(secrets.token_urlsafe(24))"

2. Environment variables 

import os
run(
    my_function,
    auth={"admin": os.environ["ADMIN_PASSWORD"]},
    secret_key=os.environ["SECRET_KEY"]
)

3. Always HTTPS (Nginx or direct SSL)

4. Restrict network access 

host="127.0.0.1"  # Localhost only (behind proxy)
host="192.168.1.100"  # Specific interface

When Is It Secure?

Setup Strong Password HTTPS Rate Limiting Verdict
Internal network ✅ ✅ ⚠️ Optional ✅ Secure
Behind VPN ✅ ✅ ⚠️ Optional ✅ Secure
Public + Nginx ✅ ✅ ✅ ✅ Secure
Public without protection ✅ ❌ ❌ ❌ Vulnerable

Bottom line: Strong password + HTTPS = Secure. Rate limiting protects against bots and stolen passwords from other sites.

Examples

# Basic
run(my_func, auth={"admin": "strong_password_here"})

# Production (persistent sessions)
run(
    my_func,
    auth={"admin": os.environ["ADMIN_PASSWORD"]},
    secret_key=os.environ["SECRET_KEY"]
)

# With HTTPS
run(
    my_func,
    auth={"admin": os.environ["ADMIN_PASSWORD"]},
    ssl_keyfile="./key.pem",
    ssl_certfile="./cert.pem"
)

Note: All authenticated users have full access to all functions.

Summary

✅ Strong password (16+ chars, unique) → Brute-force impossible
✅ HTTPS → Session cookies protected
✅ Rate limiting (public) → Bots blocked

Use as-is for internal networks. Add Nginx for public internet.
