Authentication

Secure your tools effortlessly with built-in authentication. Username/password login with session management.

Basic Usage

Add the auth parameter with username/password pairs: 

from func_to_web import run

def admin_dashboard(action: str, value: int):
    """Perform administrative actions"""
    return f"Action '{action}' executed with value: {value}"

run(
    admin_dashboard,
    auth={
        "admin": "secure_password_123",
        "user": "another_password"
    }
)

Users must log in before accessing the interface.

Session Management

Sessions are automatically managed with secure, signed cookies: 

run(
    my_function,
    auth={"admin": "password"},
    secret_key="your-secret-key-here"  # Optional
)

  • Auto-login: Users stay logged in (default: 2 weeks)
  • Logout: Button automatically appears in the header
  • Stateless: No database required

Multiple Functions

Protect multiple tools with a single login: 

from func_to_web import run

def payroll(): 
    return "Payroll Data"

def reports(): 
    return "Q4 Reports"

run(
    [payroll, reports],
    auth={
        "admin": "admin_pass",
        "analyst": "finance_pass"
    }
)

Once logged in, users access all functions.

Security Features

Built-in protections:

  • ✅ Timing attack protection: Uses secrets.compare_digest
  • ✅ Signed sessions: Cryptographically signed, tamper-proof cookies
  • ✅ CSRF protection: SameSite='Lax' cookies block cross-site requests
  • ✅ Secure by default: Sessions reset on restart (if no secret_key)

Current limitations:

  • ⚠️ No rate limiting: Vulnerable to brute-force without external protection
  • ⚠️ No audit logs: Doesn't track which user did what
  • ⚠️ Simple authentication: No 2FA, password complexity rules, or expiration

Deployment Options

Option 1: Internal Network (Simplest)

Best for: Company intranet, office network, trusted environment 

run(
    my_function,
    host="0.0.0.0",  # Accessible on local network
    auth={"admin": "strong_password_here"},
    secret_key="persistent-secret-key"
)

Security level: ✅ Good (attackers must be inside your network)

Option 2: Behind VPN (Recommended for Remote)

Best for: Remote teams, distributed organizations 

# Only accessible via Tailscale/WireGuard IP
run(
    my_function,
    host="100.64.x.x",  # Tailscale IP
    auth={"admin": "strong_password"},
    secret_key=os.environ["SECRET_KEY"]
)

Security level: ✅✅ Excellent (VPN handles network security + auth)

Option 3: Public Internet with Reverse Proxy (Advanced)

Best for: Public access with proper infrastructure

Deploy behind Nginx + Fail2Ban to add rate limiting: 

# /etc/nginx/sites-available/myapp
server {
    listen 443 ssl http2;
    server_name myapp.example.com;

    ssl_certificate /etc/letsencrypt/live/myapp.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.example.com/privkey.pem;

    location / {
        proxy_pass http://localhost:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;

        # Rate limiting
        limit_req zone=login burst=5 nodelay;
    }
}

# Rate limit config in nginx.conf
http {
    limit_req_zone $binary_remote_addr zone=login:10m rate=10r/m;
}

# Install and configure Fail2Ban
sudo apt install fail2ban

# /etc/fail2ban/jail.local
[nginx-limit-req]
enabled = true
filter = nginx-limit-req
action = iptables-multiport[name=ReqLimit, port="http,https"]
logpath = /var/log/nginx/error.log
findtime = 600
bantime = 7200
maxretry = 10

# Your app runs on localhost only
run(
    my_function,
    host="127.0.0.1",  # Only accessible via Nginx
    port=8000,
    auth={"admin": os.environ["ADMIN_PASSWORD"]},
    secret_key=os.environ["SECRET_KEY"]
)

Security level: ✅✅ Good (HTTPS + rate limiting + IP blocking)

Option 4: Behind Identity Provider (Enterprise)

Best for: Large organizations with existing SSO

Use Cloudflare Access or Google Identity-Aware Proxy to handle authentication: 

# No auth needed - IAP handles it
run(my_function, host="127.0.0.1", port=8000)

Configure IAP to allow only authorized users, then proxy to your app.

Security level: ✅✅✅ Excellent (Enterprise-grade authentication)

Security Best Practices

1. Use Strong Passwords

# ❌ Bad
auth={"admin": "admin"}

# ✅ Good
auth={"admin": "Xk9$mP2#vL8@qR5^"}

2. Use Environment Variables

import os
from func_to_web import run

run(
    my_function,
    auth={
        "admin": os.environ["ADMIN_PASSWORD"],
        "user": os.environ["USER_PASSWORD"]
    },
    secret_key=os.environ["SECRET_KEY"]
)

# Generate strong secret key
python -c "import secrets; print(secrets.token_hex(32))"

# Set variables
export ADMIN_PASSWORD="your-strong-password"
export SECRET_KEY="generated-secret-key"
python main.py

3. Enable HTTPS

# Development
run(my_function, auth={...})

# Production with self-signed cert
run(
    my_function,
    auth={...},
    ssl_keyfile="./key.pem",
    ssl_certfile="./cert.pem"
)

# Production with Let's Encrypt (via Nginx - recommended)
# Let Nginx handle SSL, app runs on localhost

4. Restrict Network Access

# ❌ Exposed to internet
run(my_function, host="0.0.0.0", auth={...})

# ✅ Localhost only (behind proxy)
run(my_function, host="127.0.0.1", auth={...})

# ✅ Specific network interface
run(my_function, host="192.168.1.100", auth={...})

5. Use Firewall Rules

# Allow only specific IPs (company VPN)
sudo ufw allow from 10.0.0.0/8 to any port 8000

# Allow only via reverse proxy
sudo ufw allow 443/tcp  # HTTPS
sudo ufw deny 8000/tcp  # Block direct access

When Is It Secure Enough?

✅ Safe to use built-in auth alone: - Internal company network - Behind corporate firewall - VPN-only access - 5-50 trusted users - Low/medium security requirements

⚠️ Need additional layers: - Exposed to public internet - Sensitive data (financial, medical, PII) - Compliance requirements (SOC 2, HIPAA) - Hundreds/thousands of users - High-value targets

)

Note: Currently, authentication is binary (Access vs No Access). All authenticated users can see and execute all functions.

Summary

The built-in authentication is secure for internal tools and can be made reasonably secure for public access with proper infrastructure:

  • Internal network: Use as-is ✅
  • VPN access: Use as-is ✅✅
  • Public internet: Add Nginx + rate limiting ✅✅
  • Enterprise: Use IAP/SSO ✅✅✅

The key is layered security: authentication + network controls + rate limiting.
